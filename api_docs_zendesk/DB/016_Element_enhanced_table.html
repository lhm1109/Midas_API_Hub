Element API - Enhanced Schema Specification

API Endpoint: {base url} + db/ELEM

Methods: POST, GET, PUT, DELETE

Schema Type: Single Entity Schema (validates one Element object)

Enhanced Schema Philosophy

Standard JSON Schema Validation + x-* Extensions for Complex Logic

  • STD Standard validators enforce: required fields, if/then logic, fixed enums, type checks
  • APP Application layer enforces: type-dependent enums, node counts, value sign constraints

Validation Architecture

✅ Standard JSON Schema Validation (~70%)

Enforced by standard validators (Ajv, etc.):

  • Base Required: TYPE, MATL, NODE
  • Conditional Required:
    • SECT required for all types except SOLID
    • STYPE required for TENSTR, COMPTR, PLATE, WALL, PLSTRS
    • WALL, W_CON required when TYPE=WALL
    • CABLE, NON_LEN required when TYPE=TENSTR AND STYPE=3
  • Fixed Enums: TYPE, CABLE, W_CON, W_TYPE
  • Array Bounds: NODE minItems=2, maxItems=8

⚠️ Application Layer Validation (~30%)

Requires custom logic using x-* metadata:

  • Type-dependent STYPE enum: Different valid values per TYPE (x-enum-by-type)
  • Exact NODE count: Specific counts per TYPE (x-node-count-by-type)
  • Value sign constraints: TENS/T_LIMIT positive/negative checks (x-value-constraint)
  • UI visibility: Conditional field display (x-ui.visibleWhen)

Transport Metadata

Schema validates: Single Element object (e.g., { "TYPE": "BEAM", "MATL": 1, ... })

API Request format:

{
  "Assign": {
    "1": { "TYPE": "BEAM", "MATL": 1, "SECT": 1, "NODE": [1, 2] },
    "2": { "TYPE": "WALL", "MATL": 1, "SECT": 1, "NODE": [3,4,5,6], "STYPE": 1, "WALL": 1, "W_CON": 0 }
  }
}

x-transport metadata:

  • URI: db/ELEM
  • Methods: POST, GET, PUT, DELETE
  • Body Root: "Assign"
